# ESUI — AI Orchestration Layer

The orchestrator is the brain. Every AI call passes through it. It handles mode selection, model routing, memory retrieval, prompt assembly, streaming, and post-call memory writes.

This doc specifies how it works.

---

## 1. The two modes

Mode is more than a system prompt. It changes:
- which model is preferred for the task
- how retrieved memory is weighted
- which tools are exposed to the AI
- the voice and structure of replies

### Ulzii Mode — the intellectual partner

Named after Ulzii (Badrushk's real name). Activates when Esui wants to **understand** something deeply.

**Disposition.** Theory-of-knowledge analyst. Learning architect. Concerned with epistemic structure: what is known, how it is known, what depends on what. Patient. Asks Socratic questions when confusion is productive. Maps territory before traversing it.

**System prompt skeleton:**

```
You are Ulzii — Esui's intellectual companion inside ESUI, her private workspace.

Your role is to help her understand, not to give her answers. You think like a
theory-of-knowledge analyst and a learning architect. You see ideas as networks:
what depends on what, what is foundational, what is downstream, where the
load-bearing assumptions live.

Speak in clear, structured prose. Build small epistemic maps before diving in.
When she is confused, find the highest-leverage misunderstanding and address it
directly. Use diagrams and concept hierarchies when they reveal structure that
prose hides. Quote primary sources when precision matters.

Tone: calm, warm, intellectually serious. Never preachy. Never condescending.
Match her register — she is sharp and reads philosophy in two languages.

Esui's boyfriend Badrushk also uses this workspace. Be warm to him; treat them
as a unit when context suggests it.

You have access to her memory and her vault. Use them. Cite when you draw on
them. Do not invent sources.

{retrieved_memories}

{pinned_context}
```

**Retrieval bias.** Boost memories with `category IN ('fact_about_user', 'fact_about_world', 'preference')` and Vault chunks tagged `research|reference|note`. Suppress recent decision/project-state memories unless the user explicitly references them.

**Preferred models.** Long-context, deep reasoning. Default to Claude Opus 4.7 with extended thinking for high-effort questions. Sonnet 4.6 for shorter exchanges.

**Tool palette.** `retrieve_more`, `cite`, `pin_to_vault`.

### Obama Mode — the strategist

Named after Obama (Badrushk's nickname). Activates when Esui wants to **build** or **decide** something.

**Disposition.** Founder. Strategist. Concerned with leverage: what moves the needle, what is the smallest experiment, what scenario should we plan against. Compresses analysis into actions. Three-scenario thinking (conservative, base, aggressive) is the default frame for major decisions.

**System prompt skeleton:**

```
You are Obama — Esui's strategic co-pilot inside ESUI, her private workspace.

Your role is to help her make things happen. You think like a founder and a
competitive strategist. You're concerned with leverage, not completeness.
What moves the needle? What's the smallest experiment that resolves the
biggest uncertainty? What scenarios should we plan against?

Default frame for big decisions: conservative / base / aggressive — three
short scenarios, each with the assumption that swings it, plus a clear
recommendation. Don't list options without a recommendation.

Be direct. Lead with the recommendation. Skip the preamble. Quantify when
useful, hand-wave when premature precision is fake. Tradeoffs are the unit
of analysis, not options.

Tone: warm, decisive, not preachy. You are her partner in building, not a
consultant. Match her register.

Her boyfriend Badrushk builds with her and runs this workspace. Be warm to
him; treat them as a unit when context suggests it.

You have access to her memory and her vault. Use them. Cite when you draw on
them. Do not invent sources or numbers.

{retrieved_memories}

{pinned_context}
```

**Retrieval bias.** Boost `category IN ('decision', 'project_state', 'goal')` and Vault chunks tagged `draft|plan|research`. Boost recency.

**Preferred models.** Fast, decisive. Default to Sonnet 4.6 for chat. Opus 4.7 for big strategic questions or competitive analysis. Gemini 3.1 Pro for market research with web grounding.

**Tool palette.** `retrieve_more`, `cite`, `pin_to_vault`, `web_search` (Gemini-only).

### Mode selection
- User toggles in UI; defaults to the user's `default_mode` preference (Ulzii initially).
- Mode is per-message, captured in `messages.mode`. Switching mid-conversation is allowed and visible in the transcript.
- Some workflows force a mode. `exam.cheatsheet` defaults to Ulzii. Together Photos doesn't use modes.

We use **LiteLLM** as the single client across all providers.

### Routing logic

```python
def select_model(task: TaskKind, mode: Mode | None, hints: dict) -> str:
    # 1) explicit override
    if hints.get("model"):
        return hints["model"]

    # 2) intent override (Obama Mode + market_research → Gemini for web grounding)
    if task.startswith("chat.") and mode == "obama" and hints.get("intent") == "market_research":
        return "gemini"

    # 3) effort scaling on chat
    if task.startswith("chat.") and hints.get("estimated_effort") == "high":
        return "opus"
    if task.startswith("chat.") and hints.get("estimated_effort") == "low":
        return "sonnet"

    # 4) baseline from task table
    return TASK_DEFAULT_MODEL[task]
```

`estimated_effort` comes from a tiny pre-call classifier (Haiku, ~30ms) that reads the user's message and returns `low | med | high`. Skipped entirely for messages under 200 chars (always `low`).

### Fallback policy

LiteLLM's built-in fallbacks handle 5xx and timeouts. Fallback events are logged to `ai_calls` with `model_id` of the fallback and a small UI indicator surfaces it ("rerouted via sonnet"). After 3 consecutive fallbacks for the same primary in 5 min, primary is auto-quarantined for 10 min.

## 3. Prompt caching

Anthropic's prompt cache is the difference between this feeling fast and feeling slow.

**Cache layers**, in order, marked with `cache_control: { type: 'ephemeral' }`:

1. **Mode system prompt** — always cached. ~5 min TTL. Same across all of a user's chat turns in the mode.
2. **Pinned conversation context** — cached when present. Invalidates on edit.
3. **Retrieved memory block** — cached when retrieval is stable across consecutive turns (we hash the retrieval block; if it's unchanged from the previous turn, reuse the same cache breakpoint).
4. **Conversation history (sliding window)** — cached up to and including the previous assistant turn. The new user turn is the only un-cached portion.

Target: >80% cache hit rate on multi-turn chat.

For Gemini and Kimi, no native prompt cache — we just don't cache, and rely on smaller injected context. Gemini context caching API can be added later.

## 4. Retrieval pipeline

Every chat turn (and most exam/vault generations) gets a retrieval block injected into the prompt.

### Inputs
- `query` — the latest user message (or generation prompt for exam)
- `user_id` — owner scope
- `mode` — for retrieval bias
- `conversation_id` — for in-conversation recency
- `pinned_doc_ids` — anything the user pinned as project context

### Algorithm

1. **Embed the query** with Voyage (`voyage-3`, 1024-dim).
2. **Multi-source vector search** — run cosine top-k on:
   - `memories` (k=20) where `owner_id = user_id AND superseded_by IS NULL AND NOT forgotten`
   - `vault_chunks` (k=20) where document `owner_id = user_id` OR `shared = true AND owner = partner`
   - `file_chunks` (k=10) — only files attached in this conversation or pinned
   - `message_embeddings` (k=10) — restricted to this conversation, last 90 days
3. **Merge + dedupe** by source_id, then **rerank** with weighted score:

```
score = 0.40 * cosine_similarity
      + 0.20 * mode_bias            # category match × 1.2, otherwise 1.0
      + 0.15 * recency_decay        # exp(-age_days / 30)
      + 0.15 * salience             # 0..1
      + 0.10 * pin_boost            # 1 if pinned else 0
```

Mode-specific overrides:

```python
weights_ulzii = {sim: 0.40, mode: 0.25, recency: 0.10, salience: 0.15, pin: 0.10}
weights_obama = {sim: 0.35, mode: 0.20, recency: 0.25, salience: 0.10, pin: 0.10}
```

4. **Trim to budget** — 2000 tokens for chat, 4000 for exam generation.
5. **Format** as a labeled block:

```
<retrieved>
## Memories
- [pref] Esui prefers Sky Blue / Vanilla / Navy / Forest. (3 days ago)
- [goal] Studying real analysis through April. (last week)
- [decision] Chose Postgres + pgvector over Pinecone for ESUI. (yesterday)

## Vault
- "Notes on Tarski's truth definitions" — § Hierarchies of Languages
  (your note, last edited yesterday)
  ...

## Files
- lecture-notes-week-4.pdf · p. 12-14: ε-δ definition...

## Recent in this conversation
- 6 turns ago you said: "I'm trying to figure out continuity in metric spaces"
</retrieved>
```

The block is built once per turn and sits in the cached system-prompt area when stable across turns.

### Stability hashing

```python
retrieval_hash = sha256(canonical_form(retrieved_items))
```

If `retrieval_hash == previous_turn.retrieval_hash`, reuse the cache breakpoint. Otherwise, invalidate this layer's cache.

## 5. Memory engine

Mem0 is configured with a custom Postgres adapter that writes to our `memories` table. Per-user namespace = `user.id`. We use Mem0 for fact extraction and storage management; retrieval is our own (it's tuned to ESUI's multi-source mix).

### Why custom adapter
Mem0's default vector backend is fine, but we want the `memories` table to be a first-class part of our schema for:
- portability (one Postgres dump exports the entire memory)
- joinability (we link memories back to source messages/docs)
- query unification (ranking memories alongside vault chunks in retrieval)

### Write path (after every chat turn)

```python
# After AI streaming completes, in a background task:
await mem0.add(
    messages=[user_msg.text, ai_msg.text],
    user_id=str(user.id),
    metadata={
        "conversation_id": str(conv.id),
        "mode": mode,
        "source_kind": "chat",
        "source_id": str(ai_msg.id),
    },
)
```

Mem0 internally:
1. Calls `memory.extract` (Haiku) with both messages plus a small context window.
2. Receives 0..N candidate facts as JSON: `[{ text, category, confidence }]`.
3. For each fact:
   - Embed with Voyage.
   - Find existing memories with cosine > 0.9.
   - If found: decide `update` (refine) vs `supersede` (contradicts) vs `noop` (duplicate).
   - Otherwise: insert.
4. Returns the new memory ids; orchestrator logs them to `ai_calls`.

### Read path

`memory.engine.retrieve(query, user_id, mode, scope_filters)` — described in §4 above.

### Consolidation (nightly Celery)

Once a day at 03:00 user-tz, for each user:
1. Pull memories from the past 24h.
2. Cluster by cosine similarity (>0.85).
3. For each cluster of size ≥ 2, ask Sonnet (`memory.consolidate`) to merge into one canonical statement, preserving newest evidence and citing all source ids.
4. Mark old members of the cluster as `superseded_by = canonical.id`.

This keeps the store from drifting into many slightly-rephrased versions of the same fact.

### Decay

`salience` is multiplied by 0.97 each day a memory goes unused (`last_used_at` not touched). When `salience < 0.1` and `last_used_at > 180d` ago, the memory is hidden from default retrieval but still queryable via `/memory`.

### Forgetting

Esui can mark any memory `forget`. Marked rows have `forgotten = true, salience = 0`. The system never re-extracts the same fact (the embedding hash is added to a per-user "do not re-learn" set in Redis, refreshed on use, expiring 90d after last_used).

## 6. Ingest pipeline (documents)

Triggered when a file is uploaded for Vault import or Exam workspace.

```
file uploaded
  ↓ (Celery task: ingest.parse_and_embed)
unstructured.io
  ↓ (sectioned blocks)
chunker (semantic, ~512 tokens, with 64-token overlap, section-aware)
  ↓ (chunks)
voyage-3 embedder (batched, 16 chunks per call)
  ↓ (vectors)
postgres (file_chunks or vault_chunks)
  ↓
classifier (Haiku) — auto-tag, classify, extract title if missing
  ↓
emit: ingest:complete via Socket.io
```

### Parse rules
- **PDFs:** unstructured.io `hi_res` strategy when document has tables/figures, else `fast`.
- **Word docs:** unstructured.io.
- **Markdown:** native parser (we own it; preserves headings as `section_path`).
- **Plain text:** split on blank lines.
- **Images:** OCR via unstructured.io if text is detected; otherwise stored as-is and only retrievable by attachment, not chunked.

### Chunking
- Target 512 tokens, hard max 768.
- Never split mid-sentence (semantic boundary detection).
- Carry parent section heading into each chunk's `section_path`.
- Overlap = 64 tokens at the head of each chunk after the first.

### Embedding
- Voyage `voyage-3`, 1024-dim, cosine.
- Batched at 16 chunks per request, ~$0.00012 per request typical.

### Failure handling
- Per-chunk embed failures get retried 3× with exponential backoff.
- Parse failures: surface in UI ("we couldn't read this file"), keep the file but mark `ingest_status='failed'`.

## 7. Streaming protocol

Server-side Socket.io events drive the chat experience. Each AI call goes:

```
client                       server                                LiteLLM/provider
  |  message:send             |
  |-------------------------->|
  |                           | persist user msg
  |                           | broadcast message:created
  |                           |---- retrieval ---->
  |                           |<-- memories       ---
  |                           |---- prompt build ---->
  |                           |
  |                           |---- stream open --------->
  |                           |<-- chunk 1 -------
  | message:ai:start          |
  |<--------------------------|
  | message:ai:delta x N      |
  |<--------------------------|
  ...
  |                           |<-- end_of_stream --
  |                           | persist final msg
  | message:ai:complete       |
  |<--------------------------|
  |                           |---- mem0.add (async) -->
  |                           |---- embed AI msg (async) -->
```

Every delta is forwarded immediately — no batching beyond what LiteLLM provides. End-of-stream triggers final persistence and post-call memory write.

If the client disconnects mid-stream, the server completes the call (don't waste tokens), persists, and the message appears whole when the client reconnects via REST history.

If the user clicks Stop, server cancels the upstream stream, persists what was generated so far with `status='complete'` if usable (≥1 sentence) or `status='error'` otherwise.

## 8. Tool use

The orchestrator exposes a small tool palette to chat AI calls. Tools are defined as Anthropic-style tool specs and translated for other providers when used there.

```yaml
tools:
  - name: retrieve_more
    description: Fetch additional retrieval hits when the initial block is insufficient.
    input_schema:
      type: object
      properties:
        query: { type: string }
        source: { type: string, enum: [vault, memories, files] }
        limit: { type: integer, default: 10 }
      required: [query, source]

  - name: cite
    description: Reference a specific chunk to support a claim. Renders as a footnote in the UI.
    input_schema:
      type: object
      properties:
        source_id: { type: string }
        source_kind: { type: string, enum: [vault, file, memory] }
        quote: { type: string }
      required: [source_id, source_kind]

  - name: pin_to_vault
    description: |
      Suggest the user save this insight as a Vault note. The user must
      confirm; you cannot bypass. The UI will show a save affordance.
    input_schema:
      type: object
      properties:
        title: { type: string }
        content_md: { type: string }
        tags: { type: array, items: { type: string } }
      required: [title, content_md]

  - name: web_search   # Obama Mode + Gemini only
    description: Perform a fresh web search via Gemini's grounding.
    input_schema:
      type: object
      properties:
        query: { type: string }
      required: [query]
```

The AI is given **read access** to memory + vault by default (via the retrieved block); tools are for second-stage access. The AI never has write access to Exam, Together, or Signals — those are user-driven flows.

## 9. Cost shape (rough)

| Operation                                    | Approx cost  |
|----------------------------------------------|--------------|
| Chat turn (Sonnet, cached, 10K in / 600 out) | ~ $0.012     |
| Chat turn (Opus, cached, 15K in / 1.5K out)  | ~ $0.10      |
| Cheatsheet (Opus, 20K in / 4K out)           | ~ $0.36      |
| Memory extract per turn (Haiku, 2K / 200)    | ~ $0.0008    |
| Embedding (voyage-3, ~1K tokens)             | ~ $0.00012   |
| Together composite (Stability + Remove.bg)   | ~ $0.10–0.20 |

At 100 chat turns/day mixed mode, expect ~$3–5/day. Documents and exam artifacts are spiky but bounded by upload + click rates. `ai_calls` table tracks every charge.

A daily soft cap of $20/user is enforced server-side; exceeding it surfaces a calm system notice and downgrades chat default to Sonnet for the rest of the day.

## 10. Putting it all together — a chat turn end-to-end

```python
async def handle_message_send(payload: SendMessage, user: User, sock: Socket):
    conv = await db.get_conversation(payload.conversation_id, user)
    user_msg = await db.create_user_message(conv, user, payload)
    await sock.broadcast_room(conv.id, "message:created", user_msg)

    mode = payload.mode or user.default_mode

    # 1. effort classify (skip if short)
    effort = "low"
    if len(user_msg.text) > 200:
        effort = await orchestrator.classify_effort(user_msg.text)

    # 2. intent classify (Obama + research?)
    intent = await orchestrator.classify_intent(user_msg.text) if mode == "obama" else None

    # 3. select model
    task = f"chat.{mode}.{'deep' if effort=='high' else 'quick'}"
    model = orchestrator.select_model(task, mode, {"intent": intent, "estimated_effort": effort})

    # 4. retrieve
    retrieved = await memory.retrieve(
        query=user_msg.text, user_id=user.id, mode=mode,
        conversation_id=conv.id, pinned_doc_ids=conv.pinned_doc_ids
    )

    # 5. build prompt with cache breakpoints
    prompt = orchestrator.build_chat_prompt(mode, conv, retrieved, recent_messages=...)

    # 6. open AI message placeholder, broadcast start
    ai_msg = await db.create_streaming_ai_message(conv, mode, model.id)
    await sock.emit_room(conv.id, "message:ai:start",
                         {"message_id": ai_msg.id, "mode": mode, "model_id": model.id})

    # 7. stream
    full_text = []
    async for delta in orchestrator.stream(model, prompt, tools=...):
        if delta.kind == "text":
            full_text.append(delta.text)
            await sock.emit_room(conv.id, "message:ai:delta",
                                 {"message_id": ai_msg.id, "delta_text": delta.text})
        elif delta.kind == "tool_use":
            result = await orchestrator.run_tool(delta.tool, delta.args, user, conv)
            await sock.emit_room(conv.id, "message:ai:tool_use", {...})
            # feed result back into stream

    # 8. finalize
    await db.complete_ai_message(ai_msg, "".join(full_text), tokens=..., cost=...)
    await sock.emit_room(conv.id, "message:ai:complete", {...})

    # 9. async post-processing
    enqueue("memory.add", user_id=user.id, conversation_id=conv.id,
            user_text=user_msg.text, ai_text="".join(full_text))
    enqueue("embed.message", message_id=ai_msg.id)
    if conv.message_count == 3 and not conv.title:
        enqueue("conversation.auto_title", conv_id=conv.id)
```

This is the canonical control flow. Every other chat path (resume, branch, regenerate) is a variant of it.
